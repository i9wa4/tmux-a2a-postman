get-status — print canonical session status JSON

Usage:
  tmux-a2a-postman get-status
  tmux-a2a-postman get-status --debug
  tmux-a2a-postman get-status --help

Output:
  JSON session status contract.
  Use nodes[*].visible_state for per-node state, queues for backlog counts,
  and compact for the compact display token.
  schema_version is 4. visible_state and compact remain stable summary fields.
  Reply-aware fields include input_required_count, waiting_on_input_count,
  and info_unread_count. nodes[*].flow.input_requests includes input_required
  and waiting_on_input detail arrays for open input requests. Details
  include direction, message_id, input_request_id, sender, recipient,
  reply_policy, opened_at, opened_at_source, and read_at when available.
  Use inspect-input --id <message_id-or-input_request_id> to inspect a specific
  open input request without reading inbox mail.
  Contextual severity fields include severity, severity_source,
  severity_reason, compact_severity, delivery, nodes[*].node_local,
  nodes[*].flow, and nodes[*].queues.
  Severity ranks from least to most urgent:
  ok, working, expected_wait, needs_action, blocked, attention_stale,
  delivery_stuck, delivery_failure.
  delivery.state distinguishes ok, queued, delivery_stuck, delivery_failure,
  and unknown. Pending post delivery becomes delivery_stuck after 180 seconds.
  nodes[*].flow separates expected waits from required action and blocked
  reports. Blocked reports are proven from structured blocked_report metadata
  and inferred from an exact first-line BLOCKED: report.
  nodes[*].screen_progress provides non-content pane evidence:
  evidence_state, last_capture_at, last_screen_change_at, and
  screen_fingerprint. Raw pane text is not included.

Debug diagnostics:
  --debug adds runtime_diagnostics as an optional object without changing
  schema_version. It requests a point-in-time daemon runtime snapshot with Go
  memory, GC, goroutine, count-only daemon cardinality fields, and
  daemon_submit queue health. daemon_submit includes worker_limit,
  active_worker_count, active_request_count, pending_request_count,
  oldest_pending_age_seconds, claimed_request_count,
  oldest_claimed_age_seconds, late_response_count,
  oldest_late_response_age_seconds, saturation_count, and last_saturated_at
  when present. This is not a persisted time series. It omits mailbox body
  content, pane content, local absolute paths, per-message identifiers, and
  unbounded lists.

  The daemon also writes the same class of scalar memory and queue counters to
  postman.log at startup and every 10 minutes as
  component=daemon_runtime event=memory_snapshot source=passive_log. Those
  passive log lines are the normal first signal for memory optimization
  analysis; capture-profile remains an explicit follow-up diagnostic.
