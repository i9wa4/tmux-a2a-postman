send-heredoc — compose and deliver a message with a quoted heredoc

Usage:
  tmux-a2a-postman send-heredoc --to <node> <<'POSTMAN_BODY'
  message text
  POSTMAN_BODY
  tmux-a2a-postman send-heredoc --help

Flags:
  --to <node>             Recipient node name (required)
  --no-reply              Mark this message as not requiring a reply
  --reply-required        Mark this message as requiring a reply
  --reply-to <message>    Message id for fallback closure or traceability
  --fills-input-request-id <id>
                          Exact input request id this message fills

Body input:
  Use a quoted heredoc delimiter:

    tmux-a2a-postman send-heredoc --to worker <<'POSTMAN_BODY'
    literal $(not executed), `backticks`, and $HOME
    ```sh
    echo "$HOME"
    ```
    POSTMAN_BODY

  The quoted delimiter preserves command substitutions, backticks, variables,
  quotes, and code fences as literal message text. Do not pass message text as
  argv. Do not use file-body or generic stdin shortcuts.

Output:
  Always JSON.
  {"sent":"filename.md","status":"processed","context_id":"...","session":"...","from":"sender","to":"recipient","reply_policy":"none","submit_path":"daemon-submit"}
  Reply-required sends include "input_request_id".
  Exact replies include "fills_input_request_id".
  Fill/reply sends may also include:
    "fill"                  requested input request, closed state, and reason
    "required_input"        current open required input for the sender
    "notice"                visible warning when terminal-looking text leaves
                            required input open
    "suggested_next_action" safe command prefix when an open request is known

Status:
  processed               Daemon-side handling was observed
  queued                  Local handoff succeeded, but daemon handling was not observed

submit_path:
  daemon-submit           Request/response path owned by the running daemon
  post                    Direct write to the session post/ queue

Timeouts:
  A daemon-submit timeout does not prove failure. The daemon may still commit a
  side-effecting send after the CLI stops waiting. Do not retry blindly; inspect
  status, recipient inbox/read evidence, or recipient-side confirmation before
  resending.
