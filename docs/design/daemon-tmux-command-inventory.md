# Daemon tmux Command Inventory

Last audited: 2026-05-08.

This document records every hardcoded production path that can execute `tmux`
from the daemon/runtime surface. The operating policy is conservative: daemon
tmux calls must stay minimal, intentional, measured, and centralized enough that
future changes can reason about purpose and cadence before adding load to the
tmux server.

Test-only fake `tmux` scripts are excluded from the inventory table. User-owned
shell template expansion in `internal/template/template.go` is also excluded
because it is not a hardcoded tmux command, but it remains a trust boundary and
must not become a hidden daemon tmux automation path.

## Policy

- New daemon-side `tmux` executions must document command family, purpose,
  trigger, expected cadence, and expected cost in this file.
- Prefer reusing the daemon's existing node/session snapshots over starting a
  fresh `tmux` process from a secondary loop.
- Keep polling loops separate by cost: session inventory, pane/node discovery,
  pane content capture, and delivery keystrokes are different workloads.
- Favor event-driven invalidation or cached snapshots before adding another
  periodic call.
- Add or update fake-`tmux` tests for new command families or new hot-loop
  behavior.
- Fixed tmux command execution should move toward a small shared runner with
  timeout, logging, and test hooks. Today only discovery has a local runner
  abstraction; most other paths call `exec.Command` directly.

## Current Hot Spots

The current daemon has four regular sources of tmux load:

1. `session_scan_interval_seconds` lists sessions every `0.10s` by default.
2. `scan_interval_seconds` performs the full daemon scan every `1s`.
3. `pane_capture_interval_seconds` performs pane discovery and screen capture
   every `5s` by default.
4. The health relay refreshes session health after selected daemon events. Since
   `handleScanTick` emits `node_activity_update` every full scan, this can add
   per-session health collection at roughly the full-scan cadence.

The most expensive repeated patterns are the duplicate global `list-panes`
calls, the per-inactive-pane `display-message "#{window_active}"` calls, the
per-node `capture-pane` calls, and the health relay's per-session
`DiscoverNodesWithCollisions` plus `list-windows`/per-window `list-panes`.

## Existing Test Coverage

The audited command families are mostly covered with fake `tmux` scripts in unit
tests rather than by a shared runner. Notable coverage includes daemon
runtime/session scan tests, idle pane-capture tests, notification delivery
tests, startup activation tests, session-health tests, and config/session-owner
tests. When a command family moves into a new daemon hot path, add a test that
asserts the expected `tmux` call count or sequence.

## Known Follow-ups

- Make the health relay consume the daemon's shared node snapshot and projected
  pane activity instead of rediscovering global nodes for each session.
- Stop refreshing session health on every `node_activity_update` unless the
  activity payload actually changed or the TUI/projection needs a fresh sample.
- Share the lightweight session-scan result with full-scan status updates so
  `handleScanTick` does not need its own `DiscoverAllSessions` call.
- Batch pane visibility by including window visibility in one pane/window
  query, or cache inactive-pane window state to remove the O(inactive panes)
  `display-message` loop.
- Reuse the full-scan node snapshot in pane capture where possible, with a
  documented staleness bound, to avoid an extra global discovery call every
  capture interval.
- Add a shared tmux runner for fixed commands with timeout, structured logging,
  and fake-runner test hooks. Most current calls use plain `exec.Command` and do
  not enforce `tmux_timeout_seconds`.
- Keep `pane_capture_max_panes` documented and tested as the backstop for large
  sessions; consider change-driven capture if tmux activity data is sufficient.
